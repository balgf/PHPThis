# Testing contract

The repository currently uses a zero-dependency test runner in `tests/run.php`.

Routing tests must prove literal dispatch with empty `PathParameters`; typed trailing-segment matches for `1` and `PHP_INT_MAX`; exact lowercase UUID versions 1 through 8 with RFC variants and explicit Nil, Max, version, variant, uppercase, wrapper, compact, length, and encoding rejection; exact lowercase ULID minimum, maximum, alphabet, length, overflow, uppercase, and encoding behavior; rejection of invalid parameter names, zero, leading zeroes, signed values, whitespace, decimal and exponent forms, Unicode digits, overflow, empty and extra segments, and URL-encoded spellings; literal precedence; startup rejection of every differing sibling type and inconsistent names across methods sharing a prefix; correct 404, 405, allowed-method, wrong-accessor, and no-fallback behavior; zero handler and database work for malformed identifiers; and unchanged `RequestHandler::handle(Request): Response` delivery through an immutable `Request` copy. Route-specific code immediately converts every validated value into a concrete identifier before narrower domain or database work. Route-scaling evidence must cover all four fixed types in large literal and typed tables and prove request-time lookup does not scan either table.

An application-owned request-handler decorator proof must show the final application class and exactly one downstream `RequestHandler`, explicit route-local construction, the complete unrolled outer-to-inner order, zero-or-one downstream invocation with the identical immutable `Request`, short-circuit and delegated outcomes, unchanged exception propagation, and exact explicit immutable `Response` replacement when present. Bound and assert every named decorator-owned side effect, including its order relative to delegation. Prove that the decorator neither wraps nor takes responsibility from `Application`, `RequestBoundary`, the terminal coordinator, `ResponseEmitter`, typed session services, response-specific cache policy, action-specific request policy, or terminal observability.

An application that adopts WebSockets must test its parser, current authentication and authorization, finite dispatch, typed operation delivery, encoder, redacted connection summary, and every recorded message, rate, connection, send, idle, heartbeat, lifetime, and shutdown bound. Real child-process and socket evidence must cover readiness without a blind sleep, accepted and denied handshakes, ordered success, malformed and oversized traffic, transport close behavior, backpressure, reconnect without unrecorded replay, occupied-listener failure, signal shutdown, bounded process exit, and cleanup in `finally`. Assert that no frame becomes a PHPThis HTTP `Request` or `Response`, and record the exact tested topology without generalizing it to production TLS, proxies, supervisor recovery, public capacity, scaling, or availability.

Every behavior test should name one outcome and arrange dependencies directly. For database handlers, include:

- expected rows or affected count;
- expected query count;
- a small fixture and a materially larger fixture;
- an assertion that both fixture sizes execute the same number of statements;
- an assertion that query fingerprints do not repeat unexpectedly;
- the query-budget failure path;
- adversarial strings passed as bound data and retrieved or matched as data rather than SQL structure;
- rejection of unknown structural selectors and oversized bounded-list shapes before database work.

For the ADR 022 finite collection proof, also cover both sort directions, cursor present and absent, equal-rank binary-key tie breaks, omitted versus direct and parsed `['']` empty selection, every supported one-to-three-category statement shape, duplicate and oversized category rejection, and SQL-looking values that remain bound data. Every accepted non-empty statement shape stays at one statement across small and materially larger SQLite fixtures; the empty selection uses zero protected statements. State that this is neither snapshot pagination nor universal authorization or injection proof.

`composer test:database-drivers` certifies the narrow PDO transport boundary. It defaults to SQLite; the dedicated CI job sets `PHPTHIS_DATABASE_TEST_DRIVERS=sqlite,mysql,pgsql` and supplies real services. Do not treat an unconfigured driver as a pass or fold driver-specific SQL into a runtime dialect abstraction.

Inspect the in-memory query trace directly and print its redacted JSON snapshot only when useful for a focused failure. Do not write every query during the full test suite.

ADR 023 evidence must cover success, mapped failure, known denial where applicable, unknown failure, exactly 32 lowercase-hex correlation characters, the same `X-Request-ID` response value, zero and multiple distinct finite database sources, aggregate and per-source duplicate-query and budget-overrun values, trace truncation, status-only denials, class-only unknown failures, omission of all request and SQL values, exactly one sink invocation attempt, and an unchanged selected response when the sink throws. The test proves the invocation attempt, not durable delivery or network emission.

An application that adopts server-side data caching must test miss and hit paths, finite expiry, malformed or version-incompatible values, backend failure, tenant separation, invalidation after committed writes, a miss racing a committed write, and its recorded stampede/concurrency policy. Compare small and large fixtures with an empty cache so a warm result cannot hide N+1 behavior. Cache-operation counts and bounded redacted cache evidence remain separate from `QueryBudget` and `QueryTrace`. Assert exact key grammar with synthetic non-sensitive values; never copy production or sensitive key material into tests, and never log complete keys or values.

The ADR 028 Redis proof additionally tests two distinct Redis process identities, canonical cache-value encoding, source and cache-write failure, stale-refill expiry recovery, authoritative commit before invalidation, nonblocking `SET NX PX` acquisition, explicit owner-checked renewal and release, contention, stale owners, backend outage, process termination, and post-expiry acquisition. Real-backend evidence proves renewal, stale-owner rejection, and server rejection against `ext-redis ^6.3`, Redis server `>=7.4` and `<9.0`, cache `127.0.0.1:6379/0`, and lease `127.0.0.1:6380/0` by default. Source is under `example/`; evidence is under `tests/`. Do not infer fencing, failover, replication, restart, partition, production pause, or exactly-once guarantees from deterministic tests.

An application that adopts ADR 024 durable jobs must test commit-visible publication and rollback exclusion, bounded envelope parsing and finite dispatch, idle and successful one-shot outcomes, exact finite retry delays from freshly observed failure time, completion rollback when handler time reaches lease expiry, lease expiry and stale-token fencing, terminal attempt and expired-final-lease handling, poison dead letters, idempotent duplicate delivery, real subprocess termination and post-expiry recovery, one delivery per fresh process, output and durable-state redaction, and constant statement counts across small and materially larger queues. Unroll repeated invocations or use fresh processes; never hide database polling in a loop.

An application that adopts ADR 025 CLI or scheduling must execute its real console in fresh subprocesses and test the complete finite command and argument grammar, rejection before application I/O, exact exit codes, stdout/stderr exclusivity and bytes, every expected outcome, operational and unexpected failure redaction, explicit-clock cadence boundaries, not-due exclusion from scheduled work, nonblocking overlap, owned cleanup or release, one-pass bounds, fresh HTTP and CLI composition, and every recorded missed-run, catch-up, repeated-slot, supervisor, and topology limit. For the ADR 028 Redis proof, also prove finite owner-token expiry and renewal, stale-owner rejection, fail-closed backend errors, process termination, and post-expiry acquisition. Do not mock a generic console or scheduler, hide repeated work in a loop, describe a same-host file lock as distributed coordination, or describe an expiring lease without fencing as absolute exclusion.

An application that adopts ADR 027 migrations must execute the real console in fresh subprocesses and prove an empty SQLite database applies the exact finite manifest order, the complete bounded ledger has the expected identifiers, positions, checksums, and valid recorded timestamps, an unchanged rerun returns `up_to_date` without executing a pending manifest step, edited content and malformed or overflowing history fail before pending work, same-host lock contention is nonblocking and changes no database state, and one failed migration plus ledger insert rolls back while earlier committed migrations remain. Prove forward continuation, exact exits and stream bytes, complete redaction, fresh migration composition, and zero migration work during HTTP startup. Do not call database methods in a loop or generalize the SQLite and `flock` evidence to another engine or host topology.

An application that defines HTTP caching must test the exact `Cache-Control` policy and every applicable validator and `Vary` dimension, including cookie-emitting responses as a separate case. Personalized, session-affecting, authenticated, and sensitive responses begin as `private, no-store` unless an accepted application decision and tests prove a different safe policy.

A protected request-policy proof must distinguish unauthenticated, ordinary forbidden, cross-tenant, permitted, and unexpected policy-failure paths. Record the exact `authenticate -> resolve tenant -> authorize -> handler` order and assert that each failure prevents all later calls. Give any policy reads and protected handler work distinct budgets and traces; when proof policies are I/O-free, state and test that bound explicitly. Every denial proves zero protected queries and writes. Assert the generic `401` with `WWW-Authenticate: Bearer`, the shared generic `403`, `private, no-store`, status-only denial summaries, class-only unexpected-failure summaries, and absence of credentials and sensitive identifiers from bodies, headers, summaries, and trace snapshots. Replace each policy implementation independently through manual composition and prove that the permitted protected operation receives the exact concrete principal and tenant context. A concrete credential parser separately tests missing, malformed, expired, revoked, and rejected inputs.

Inbound request and command factories require adversarial tests for missing versus explicit null, unknown and secret-looking fields, booleans, integers, coercive strings, lists, objects and nested shapes, numeric overflow, malformed JSON, invalid UTF-8, excessive nesting, and exact byte, field, and item-count limits as applicable. Assert the recorded preservation, rejection, or explicit normalization policy, deterministic generic response bytes, and absence of submitted values from bodies, headers, logs, and traces. Every rejection proves no operation-owned downstream I/O or mutation; when a separate typed operation seam exists, it also proves zero seam calls. Record and separately bound any transport, session, authentication, tenant, or authorization work deliberately ordered before parsing. PHPStan verifies the concrete boundary value and any typed seam signature. Database projections retain their exact selected-column, null, runtime-type, canonical-scalar, and schema-range evidence after the query occurs.

`RequestReader` tests must cover method/path normalization, header normalization and collisions, non-string runtime values, query-key and metadata-count bounds, an exact-limit body, an oversized declared body, an oversized actual body, and a mismatched `Content-Length`. Error-boundary tests must prove exact-class mapping, public-message non-disclosure, exclusion from handler-owned work for rejected client input, explicit accounting for any session or policy work ordered earlier, and unchanged rethrowing of unknown failures.

ADR 026 multipart tests additionally cover configured disablement, POST-only input, missing/empty/duplicate/oversized boundary, disallowed unquoted boundary characters, canonical and overflowing length, total multipart cap, `Transfer-Encoding`, parsed text fields, empty/missing/nested/multiple normalized files, raw scalar duplicate collapse, unknown or wrongly typed metadata, controls, temporary-path bound, reported-size contradictions, every `RequestUploadError`, unknown runtime error, ordinary-body compatibility, and route-copy preservation. Application tests prove the exact field and file bound, rejection before storage, `is_uploaded_file`, actual-size equality on success, server-generated storage identity, client metadata non-use, verified private modes, successful move ownership, redacted unavailable storage, and complete stored bytes. They do not inject an actual-size race, kernel move/chmod failure, or cleanup failure; keep those limits explicit until production-representative fault evidence exists.

Local-file response tests cover absolute path and non-negative bytes, empty string-body exclusivity, exact canonical `Content-Length`, case-insensitive duplicate and control-bearing headers, prohibited transfer/range framing, prior-output classification, regular-file and exact-size checks before headers, exact fixed-chunk output, handle closure, path-free pre-header failures, response-copy preservation, full `200` behavior for a `Range` request, and one generic fallback only before response start. The visible post-header no-replacement branch remains pinned, but the suite does not inject a mid-read failure. Execute a real SAPI multipart/upload/download flow with client output directed to files; record the fixed emitter allocation bound separately from any deployment-specific PHP, server, proxy, TLS, or client buffering claim.

Cookie tests must prove deterministic serialization, separate repeated `Set-Cookie` fields, rejection of invalid names, values, paths, attributes, duplicate name/path scopes, and manually encoded cookie headers. Session transport tests must use an explicitly isolated native file path and cover a configured but unused stateless request, anonymous reads without storage, malformed, duplicated, attacker-selected, stale, and obsolete identifiers, every scalar snapshot bound, callback rollback, lock release, multi-operation unissued-ID cleanup, identifier regeneration, delayed-response cookie safety, unknown and finalization failure cleanup, logout invalidation, and cookie expiration. Application tests additionally prove authentication-time regeneration, idle and absolute expiry, CSRF, authorization, revocation, and concurrent-request policies when those concerns apply; each absent concern is explicitly not applicable.

A Strict Profile rule requires at least one failing fixture, one passing fixture, and assertions for its permanent identifier and source line. Type-aware fixtures run through PHPStan; syntax guard fixtures call the same token-aware implementation used by the repository guardrail. `PHT006` fixtures must reject arbitrary strings, runtime interpolation or concatenation, sanitizer-shaped returns, argument unpacking, and method-callable indirection while accepting direct constants and finite constant mappings. The installed-consumer proof must independently submit a PHT006 failure through the public checker.

The consumer proof must install a mirrored framework package into a fresh temporary project, run the public checker and application behavior tests, execute the real front controller, and submit adversarial files outside conventional source roots. It must also prove that consumer PHPStan configuration, baselines, and inline ignores cannot weaken the profile. The local package-archive proof must compare the complete Composer and Git export inventory with the explicit release allowlist. The Git export comparison requires a clean worktree; a dirty-tree Composer archive pass is useful development feedback but not release evidence. Alpha publication separately verifies the actual Packagist-preferred dist because a local archive cannot prove hosting-provider output.

The duplication advisory requires a fast direct scanner suite for its exact threshold, comment and whitespace normalization, maximal overlap and nested-copy consolidation, equal and unequal periodic copies, separated same-file and multi-location grouping, deterministic order, fixed group and location truncation, resource saturation, a near-limit 64 MiB subprocess proof, escaped bounded paths, and absence of PHP source-token values. The installed-consumer proof additionally plants a deliberate clone outside conventional roots and matching dependency, installed-framework, and VCS controls; it must prove concise normal output, bounded debug-only locations, no source-token disclosure, exit zero for duplication alone, unchanged structural and PHPStan failures, continuation through the canonical behavior-test stage, and PHPStan continuation after a fixed scan limit is reached.

An intentionally invalid performance control uses a `.php.fixture` suffix so it cannot be mistaken for accepted repository PHP. A proof must pass its source to the real Strict Profile checker, assert the stable rejection, execute it only in an isolated subprocess, and compare its output with the accepted implementation. Never exclude an invalid `.php` file from guardrails.

Do not mock fluent APIs or framework internals. Do not introduce an ORM, query builder, repository, generic paginator, SQL/binding/placeholder helper, runtime SQL generator, arbitrary SQL string, transaction callback, or dialect abstraction to make a test convenient. Prefer real value objects and an in-memory database when its behavior matches the production database feature under test. Database-specific SQL still requires integration tests against that database. Record the source and date of runtime-credential and migration-authority verification; a passing query test does not prove least privilege.
