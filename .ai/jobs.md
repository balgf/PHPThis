# Durable jobs contract

Read [ADR 024](../docs/decisions/024-application-owned-sqlite-durable-jobs.md) and [Durable jobs](../docs/jobs.md) before adding or changing deferred work. PHPThis provides no core queue, worker, job, dispatcher, scheduler, or command-map API. The accepted evidence is one application-owned SQLite recipe.

Rules:

- Insert the job row through the same `Connection`, in the same explicit SQLite transaction, as the business write that requires the job. Commit makes both visible; rollback leaves neither. Do not claim atomicity across connections, databases, brokers, or external services.
- Store an explicit bounded JSON envelope with a version, finite code-owned type, application-generated idempotency key, and operation-specific payload. Parse the stored `mixed` value once into a final readonly job value; reject missing, unknown, malformed, oversized, unsupported-version, and unsupported-type input before dispatch.
- Dispatch with a finite `match` over supported type and version combinations. Do not use discovery, reflection, class names from storage, serialized PHP objects, a service container, or an event bus.
- Claim at most one due row per process invocation. Use one complete constant SQLite `UPDATE ... RETURNING` statement with explicit named bindings, deterministic ordering, a finite lease deadline, a fresh opaque lease token, and a bounded attempt count inside one short explicit transaction. Sample the explicit clock again before handler work and after success or failure; claim time alone cannot prove continuing ownership. Acknowledge, retry, or dead-letter only while the row is leased, its identity and token match, and that freshly checked lease is still unexpired.
- Treat delivery as at-least-once. A lease may expire after work begins, and a process may stop before completion is recorded. Make the demonstrated database effect idempotent with a unique application-owned idempotency key and finalize that effect and job state in one explicit SQLite transaction.
- Give retries a finite maximum and a finite code-owned backoff schedule measured from the freshly observed failure time. A retry records only the next eligible time and a redacted diagnostic code. A malformed envelope, unsupported type or version, and an attempt that reaches the accepted maximum become terminal dead letters. The claim path must also terminate an expired final-attempt lease so a process crash cannot strand it indefinitely.
- Store and emit only finite redacted diagnostic codes and bounded operational identifiers. Never persist or emit exception messages, stack traces, SQL, bindings, envelopes, payload values, credentials, customer data, or external response bodies as job diagnostics.
- Compose the connection, clock, finite dispatcher, and handler fresh for one worker invocation; claim and finalize zero or one delivery, emit one bounded terminal result, and exit. An external supervisor may invoke the process again. Do not add an in-process worker loop, connection reuse across deliveries, hidden retry loop, or graceful-stop subsystem.
- Keep every job SQL statement at its direct `Connection` call with its explicit parameter array. Do not add an ORM, Active Record, repository, query builder, SQL/binding/placeholder helper, transaction callback, facade, global helper, generic or framework middleware abstraction, or hidden instrumentation.
- Do not claim exactly-once external effects. A provider idempotency key, durable receipt, compensation policy, timeout policy, and reconciliation process are separate application decisions for each external integration.
- Record the concrete application's one-shot command, supervisor, signals, timeout, stop behavior, and deployment topology before adoption. ADR 025 keeps command and scheduler composition application-owned: the accepted example exposes this operation only as `jobs:run-one` through its sole console, and its one-pass scheduler calls the same operation directly. Do not add a generic framework command map, scheduler, supervisor abstraction, worker discovery, or process-management API.

Tests must prove committed publication, rollback exclusion, finite envelope parsing and dispatch, success, idle behavior, bounded retry and backoff, lease expiry and redelivery, stale or expired lease fencing, maximum-attempt and expired-final-lease dead-lettering, poison-envelope dead-lettering, idempotent duplicate delivery, process termination followed by post-expiry recovery, one delivery per fresh process, complete diagnostic redaction, and constant statement bounds for every exercised transition.
