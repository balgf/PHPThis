# Application CLI and scheduler contract

Read [ADR 025](../docs/decisions/025-application-owned-explicit-cli-and-scheduler.md), [ADR 028](../docs/decisions/028-application-owned-redis-cache-and-schedule-lease.md) when distributed overlap changes, and [Application CLI and scheduler](../docs/cli.md) before adding or changing an operational application command. PHPThis provides no core CLI command, command map, argument parser, scheduler, lock, lease, daemon, or process manager. Framework `bin/phpthis` remains the installed `check` boundary only.

Rules:

- Give the application one explicitly named console entrypoint and one finite code-owned command map. Construct each command and dependency manually; do not discover commands, resolve a class name from input, use a service container, scan files, or install a generic console or scheduler facade.
- Parse `$argv` once at the console boundary. Reject an unknown command separately from invalid, duplicate, misplaced, oversized, or unsupported arguments before application I/O. Narrow accepted values into command-specific typed values; never pass a raw argument array into operation behavior.
- Record one closed exit contract. The accepted pattern uses `0` for every expected command outcome, including idle, not-due, and overlap-skipped; `2` for unknown command or invalid arguments; and `1` for operational or unexpected failure.
- Emit exactly one newline-terminated JSON object to stdout for exit `0`, with stderr empty. Emit exactly one generic newline-terminated JSON error to stderr for exits `1` or `2`, with stdout empty. Keep key order and finite outcome vocabulary stable. The current `schedule:run` success adds a bounded `coordination` list after `command` and `outcome`; a Redis operational failure adds its finite list after `error: command_failed`. Never emit submitted arguments, paths, Redis endpoints, keys, values, owner tokens, raw replies, DSNs, credentials, exception classes or messages, stacks, SQL, bindings, envelopes, payloads, or domain values.
- Compose fresh command-scoped connections, budgets, traces, clocks, and operations for every process invocation. HTTP and CLI may share immutable configuration and explicit application construction code, but never a mutable request object, request correlation, session lifecycle, connection, budget, trace, or other request-scoped state.
- Keep `jobs:run-one` a direct one-pass call to the adopted application-owned job operation. It handles zero or one delivery and exits. Do not add polling, recursion, subprocess dispatch, an internal supervisor, or another spelling for that operation.
- Keep `schedule:run` cron-friendly and one-pass. Use an explicit UTC clock and finite cadence. A due pass may invoke the exact same in-process operation as the direct command; do not create a second handler or enqueue path.
- Guard the current example's scheduled pass with its application-private Redis lease: one fresh owner token, nonblocking `SET NX PX` acquisition, fixed owner-checking renewal and release scripts, finite TTL and timeouts, and a separate `noeviction` lease process. The proof defaults cache to `127.0.0.1:6379/0` and lease to `127.0.0.1:6380/0`; logical databases on one process are not separation. Fail closed on Redis operation errors, retain their bounded coordination evidence, and treat contention as an explicit expected outcome. Do not add retry, waiting, a renewal loop, a fencing-token claim, or a generic distributed-lock API. A different application may retain ADR 025's same-host `flock` only when its recorded topology is single-host.
- Record missed-run, catch-up, repeated-in-slot, cron frequency, command timeout, Redis restart and failover behavior, TTL expiry during work, and distributed-coordination policy. The accepted Redis lease does not deduplicate sequential invocations, replace job idempotency, fence work after expiry, survive every partition, or prove an external effect executes once.
- Keep all operational commands, argument policies, output schemas, locks, clocks, schedules, side effects, and tests application-owned. Do not add them to PHPThis core or route them through `vendor/bin/phpthis`.
- Keep `database:migrate` as the sole ADR 027 migration spelling. It composes fresh separately authorized migration state, acquires its application-private nonblocking same-host lock before database work, executes the finite unrolled SQLite manifest once, and exits with `applied` or `up_to_date`. Do not invoke it from HTTP startup, dependency hooks, the scheduler, or another command.

Tests must execute the real console in fresh subprocesses and prove exact success, idle, not-due, overlap, unknown-command, invalid-argument, missing-resource, migration applied and up-to-date outcomes, finite migration failures, and unexpected-failure behavior; stdout/stderr exclusivity and exact bytes; finite argument bounds; pre-I/O rejection; explicit-clock cadence boundaries; nonblocking overlap; Redis contention, renewal, stale-owner, safe-release, outage, termination, and expiry-recovery behavior when ADR 028 is adopted; one-pass resource bounds; fresh composition; redaction; and no hidden second console, scheduler, or migration path.
