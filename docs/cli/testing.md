# CLI testing

For production adoption, execute the real application console in fresh subprocesses. Add evidence for every accepted command and applicable outcome, the complete argument grammar and byte limits, rejection before application I/O, exact exit codes, exclusive stdout/stderr bytes, stable key order, final newline, missing-resource and deliberately injected unexpected-failure behavior, and complete secrets redaction.

Use an explicit deterministic clock to exercise UTC minutes immediately before, on, and after the five-minute boundary. Prove not-due performs no scheduled operation, a held Redis lease produces immediate `overlap_skipped`, injected connection, acquisition, renewal, and release failures are generic and fail-closed, and owned release occurs after success and failure. Assert the exact bounded `coordination` list on every schedule success and Redis operational failure while preserving one-line stream exclusivity and redaction.

Prove one due pass calls the same one-job operation at most once and handles at most one delivery. Record and test that there is no catch-up or slot ledger and that sequential invocations in one due minute are not deduplicated. Verify fresh HTTP and CLI mutable state under only the recorded immutable shared configuration.

For `database:migrate`, prove fresh-database manifest order, exact bounded ledger state, unchanged no-op rerun, edited-content and malformed or overflowing ledger rejection before pending work, immediate same-host lock contention with no database change, per-migration rollback with earlier commits preserved, forward continuation, and complete output redaction. Prove ordinary HTTP startup performs no migration work. Treat SQLite and local `flock` results as application evidence only.

For the ADR 028 lease, also prove `SET NX PX` owner-token acquisition, explicit renewal, stale-owner renewal and release rejection, safe owned release, separate `noeviction` topology, backend outage, process termination, and post-expiry acquisition against the recorded Redis version. The current real-backend evidence renews TTL, rejects a stale owner after a successor acquires, and distinguishes server rejection from contention on the lease process. The focused proof does not establish fencing, failover, replication, restart, partition, pause, clock, or exactly-once behavior.

These are consumer evidence requirements. The current example proof is intentionally bounded: its real-console cases cover invalid, unknown, missing-database, `completed`, and `idle`; direct command and Redis integration tests cover deterministic cadence and coordination; ADR 024 worker tests plus exhaustive typed mapping and static analysis cover retry and dead-letter outcomes. The focused console proof does not replace the complete application gate.
