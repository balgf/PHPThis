# PHPThis 0.1.0-alpha.4

Release identity: `0.1.0-alpha.4`. Publication state is external; verify the approved tags and Packagist packages.

- Composer version: `0.1.0-alpha.4`
- Framework tag: `v0.1.0-alpha.4`
- Skeleton tag: `v0.1.0-alpha.4`

These notes describe the accepted bounded Alpha 4 claim. Identity and gated publication authorization do not announce either tag, either package, or the public installation path. Every external operation remains subject to the complete release proof in `RELEASING.md`.

## Purpose

Alpha 4 carries the complete Alpha 3 foundation forward and rolls up four bounded post-Alpha-3 changes: canonical UUID and ULID route types, the route-local application-owned request-handler decorator profile, the application-owned WebSocket integration profile, and a framework-CI PHPStan memory-limit repair.

This prerelease remains experimental evaluation software. It is not production-ready and makes no backward-compatibility promise across prereleases.

## Canonical UUID and ULID routes

ADR 032 and Consumer Contract version 8 add two fixed full-segment route types:

- `uuid` accepts only the lowercase canonical RFC-variant grammar recorded by ADR 032, with version nibbles 1 through 8, and rejects Nil, Max, uppercase, compact, braced, URN-prefixed, encoded, and other alternate spellings.
- `ulid` accepts only the lowercase 26-character canonical grammar recorded by ADR 032, including the first-character bound that keeps the value within 128 bits, and rejects uppercase and excluded Crockford characters.

`PathParameters::uuid()` and `PathParameters::ulid()` return the validated representation unchanged. Applications use the narrowest route type, immediately wrap the value in a concrete application-owned identifier, and apply any narrower domain policy before authorization, database, or external work. PHPThis adds no identifier library, generator, normalizer, automatic binding, record lookup, database cast, storage policy, or ORM behavior.

Exact literals retain precedence and the deterministic state index retains one typed transition per state. Conflicting parameter types fail at router construction; a failed UUID or ULID match does not fall back to `token`.

This bounded runtime addition raises the core-source ceiling from 2,500 to 2,600 physical lines. The reviewed core occupies 2,592 lines. The remaining eight lines authorize no adjacent route type, third parameter, parser API, binding behavior, or compatibility spelling.

## Route-local application-owned decorators

ADR 033 and Consumer Contract version 9 accept one optional application pattern without adding framework runtime behavior. A final application-owned request-handler decorator implements only the existing `RequestHandler`, wraps exactly one downstream `RequestHandler`, and owns one narrowly named route concern. Its complete outer-to-inner order remains visibly constructed beside the affected `Route`.

A decorator may short-circuit with an explicit response or call its downstream exactly once with the identical immutable `Request`. It preserves exception identity and explicitly preserves every unchanged immutable response field when replacing a response. Any decorator-owned I/O is narrowly named, bounded, visible, and tested.

This is not generic middleware. Alpha 4 adds no framework decorator class, middleware interface, pipeline, stack, registry, priority ordering, `$next` callable, context bag, discovery, automatic composition, or wrapper around `Application`, `RequestBoundary`, the terminal coordinator, or `ResponseEmitter`.

## Application-owned WebSocket profile

ADR 034 documents how an application may select and pin a mature third-party WebSocket runtime while keeping one separate visible process and composition root. Handshakes and frames remain outside PHPThis HTTP and never become PHPThis `Request` or `Response` values. Typed shared application behavior begins only after WebSocket-owned parsing, current authentication, and current authorization.

The profile requires the application to own and prove its handshake, credential, authorization, message, rate, connection, backpressure, lifetime, delivery, redaction, shutdown, supervisor, scaling, and deployment policies. The accepted independent consumer proof demonstrates that this separation is viable with one pinned runtime and one exact application recipe; its numeric limits and runtime-specific behavior are not PHPThis defaults or production guarantees.

Alpha 4 adds no framework WebSocket server, client, event loop, frame, connection manager, daemon, supervisor, gateway, channel, broadcaster, pub/sub, middleware, retry, replay, acknowledgement, reconnect, or exactly-once API and no third-party framework runtime dependency.

## Framework analysis reliability

The framework repository's canonical PHPStan command now permits 512 MiB instead of 256 MiB after the expanded framework proof surface exhausted the former limit in CI. Maximum-level PHPStan, strict rules, the analyzed source, and the pass criteria remain unchanged. This change does not weaken analysis and does not establish a consumer PHP memory requirement: Composer does not inherit dependency root scripts, and consuming applications continue to use the installed framework-owned checker contract.

## Upgrade from Alpha 3

Alpha 3 consumers move from Consumer Contract version 7 to version 9 by applying the version 8 and version 9 migrations in order.

For version 8:

1. Audit every identifier-shaped `{name:token}` route.
2. Leave genuinely opaque tokens unchanged.
3. Change a declared UUID or ULID representation to `{name:uuid}` or `{name:ulid}` and use the matching `PathParameters` accessor.
4. Immediately wrap the unchanged string in a concrete application identifier and enforce any narrower domain rule before downstream work.
5. Add canonical success, malformed, uppercase, encoded, alternate-spelling, 404, 405, overlap, wrong-accessor, and zero-downstream-work evidence.

This migration implies no data rewrite, schema change, identifier-generation policy, automatic binding, database lookup, or runtime dependency. An external integration that requires another spelling needs a separately approved boundary rather than router normalization.

For version 9, an application that does not adopt request-handler decorators changes no source. An adopting application keeps each final decorator and its one downstream handler visible at the route, removes generic pipeline, registry, context, or boundary-wrapper behavior, and adds evidence for zero-or-one downstream calls, exact request identity, unchanged exception identity, complete response-field preservation, nesting order, short circuit, and bounded named I/O.

WebSockets are optional and add no migration for an application that does not use them. An adopting application records and proves its own pinned runtime, separate process, exact policies, numeric bounds, and operational topology; it does not copy the independent proof's numbers as framework defaults.

The supported PHP range remains `~8.4.0`. Strict Profile version 2 and permanent diagnostics `PHT001` through `PHT006` remain unchanged. Do not add consumer PHPStan configuration or weaken Composer platform checks. Update to the exact Alpha 4 prerelease only after reviewing the Contract version 8 and version 9 changes, then run the complete application gate.

## Carried-forward boundary

Alpha 4 retains zero third-party framework runtime dependencies, explicit manual composition, immutable HTTP values, finite route declarations, direct visible SQL through `Connection`, bound data, compile-time-constant SQL structure, query budgets and traces, and separately certified PDO transport for SQLite, MySQL 8.4, and PostgreSQL 17.

The release does not add or permit:

- an ORM, Active Record, lazy loading, model or repository layer, query builder, generated SQL, binding or placeholder helper, generic paginator, or SQL-dialect abstraction;
- implicit or global scopes, observer magic, automatic discovery, reflection-based wiring, service container, facade, global helper, macro system, dynamic proxy, or autowiring;
- generic or framework middleware, a hidden request pipeline, request-context bag, native WebSocket runtime, real-time gateway, channel, broadcaster, or pub/sub abstraction;
- hidden database calls, database work inside loops, hidden retries, automatic cache refill or invalidation, automatic job discovery, inferred migration rollback, or exactly-once external-effect claims; or
- production readiness, a support or security-response SLA, complete CRUD, portable application SQL or DDL, deployment certification, or capacity certification.

Consumers remain responsible for their application abstractions, identity, tenancy, authorization, schema, database plans and grants, concurrency, deployment, operational limits, observability destination, third-party runtimes, and complete behavior evidence.
