# PHPThis 0.1.0-alpha.3

Release identity: `0.1.0-alpha.3`. Publication state is external; verify the approved tags and Packagist packages.

- Composer version: `0.1.0-alpha.3`
- Framework tag: `v0.1.0-alpha.3`
- Skeleton tag: `v0.1.0-alpha.3`

These notes describe the accepted bounded Alpha 3 claim. Identity and gated publication authorization do not announce either tag, either package, or the public installation path. Every external operation remains subject to the complete release proof in `RELEASING.md`.

## Purpose

Alpha 3 is a tooling-only release. It carries the complete bounded Alpha 2 foundation and consumer profile forward unchanged and adds ADR 030's report-only consumer duplication advisory to the installed `phpthis check` workflow.

This prerelease remains experimental evaluation software. It is not production-ready and makes no backward-compatibility promise across prereleases.

## Tooling addition

The application checker now inspects exact repeated PHP token runs using the same deterministic application-file manifest used by structural checks and PHPStan. Structural checks and the duplication scanner share one captured source read; PHPStan receives the same manifest paths. A possible group requires at least 48 consecutive normalized tokens after ordinary PHP opening tags, comments, docblocks, and whitespace are removed while identifiers and literal bytes remain exact.

The advisory is deliberately review-only:

- a complete no-match scan prints one concise pass line;
- a possible group prints one concise normal-mode advisory and offers bounded application-relative locations through `phpthis check --debug`;
- reaching a fixed resource bound prints an explicit incomplete advisory, while a caught failure during report generation prints one unavailable advisory;
- duplication alone returns success and never prevents PHPStan or the application-owned behavior-test stage from running; and
- output contains no PHP source snippets, normalized token text, hashes, absolute paths, or exception details.

It has no `PHT` identifier, baseline, suppression, ignore path, consumer configuration, score, automatic rewrite, or validity effect. A human remains accountable for deciding whether reported repetition is accidental, intentionally explicit, or evidence for a narrowly named operation.

## Release artifact hygiene

The repository README adds the PHPThis banner as a presentation-only change and uses an absolute repository asset URL so the Composer package does not need to carry the binary artwork. Composer and Git export policies now both exclude a root `.DS_Store`, preventing ignored host metadata from entering or invalidating a release archive. Neither change affects the framework runtime or consumer checker contract.

## Carried-forward boundary

Alpha 3 adds no framework runtime class or behavior and changes no routing, HTTP, session, response, database, cache, queue, migration, scheduling, file-transfer, authentication, authorization, or tenancy behavior. It carries forward:

- Consumer Contract version 7;
- Strict Profile version 2 and permanent diagnostics `PHT001` through `PHT006`;
- the supported PHP 8.4.x range `~8.4.0`;
- the bounded framework core and PDO transport certified separately against SQLite, MySQL 8.4, and PostgreSQL 17; and
- ADR 029's Alpha 2 capability ownership, sanitized consumer profile, explicit application patterns, and permanent no-magic boundaries.

The existing SQLite and Redis examples remain application-owned evidence rather than framework services. Database transport certification still does not make application SQL, DDL, locking, returned values, errors, or transaction behavior portable between engines.

## Upgrade and output change

An Alpha 2 consumer on PHP 8.4.x requires no source, configuration, runtime, Consumer Contract, Strict Profile, or diagnostic migration. Update to the exact Alpha 3 prerelease and rerun the complete project check.

The observable change is additional duplication-advisory output before PHPStan. Automation or tests that assert the exact `phpthis check` output stream may need to accept the new no-match, possible-group, incomplete-scan, or unavailable-scan line. A duplication advisory never changes the command's validity result by itself. Applications must not suppress it by excluding owned source or automatically generalize explicit SQL, security sequencing, finite operations, or independent tests merely to remove a report.

Debug output intentionally reveals bounded application-relative filenames and line topology. Application paths must not contain secrets or customer data.

PHP 8.5 and later remain unsupported until a separate compatibility decision and complete CI evidence exist. Consumers must not bypass Composer's platform check.

## Limitations and permanent exclusions

The advisory deliberately misses copied blocks below 48 normalized tokens, renamed identifiers or literals, reordered or semantically equivalent code, non-PHP duplication, and files skipped after a fixed resource bound. It may report intentional boilerplate or deliberately explicit code. It is a bounded review lead, not proof that an application is or is not DRY.

Alpha 3 does not add or permit:

- an ORM, Active Record, model or repository layer, lazy loading, query builder, generated SQL, binding or placeholder helper, generic paginator, or dialect abstraction;
- implicit or global scopes, observer magic, automatic discovery, reflection-based wiring, a service container, generic facade, global helper, macro system, dynamic proxy, or autowiring mechanism;
- a framework-owned authentication, identity, tenant, authorization, validation, logging, cache, queue, worker, scheduler, migration, storage, filesystem, or distributed-lock abstraction;
- hidden database calls, database work inside loops, hidden retries, automatic cache refill or invalidation, automatic job discovery, inferred migration rollback, or exactly-once external-effect claims; or
- production readiness, a support or security-response SLA, complete CRUD, Update or Delete evidence, portable application SQL or DDL, deployment certification, or capacity certification.

Consumers remain responsible for their concrete abstractions, identity, tenancy, authorization, schema, database plans and grants, concurrency, operational policy, deployment, observability destination, and complete behavior evidence.
