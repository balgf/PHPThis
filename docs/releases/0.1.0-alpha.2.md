# PHPThis 0.1.0-alpha.2

Release identity: `0.1.0-alpha.2`. Publication state is external; verify the approved tags and Packagist packages.

- Composer version: `0.1.0-alpha.2`
- Framework tag: `v0.1.0-alpha.2`
- Skeleton tag: `v0.1.0-alpha.2`

These notes describe the approved bounded Alpha 2 claim. Identity and gated publication authorization do not announce either tag, either package, or the public installation path. Every external operation remains subject to the complete release proof in `RELEASING.md`.

## Purpose

Alpha 2 extends the checked foundation with two bounded core capabilities and a set of explicit application-owned patterns. It also composes the accepted boundaries in one sanitized account-scoped request so an AI and a human reviewer can inspect policy order, input parsing, raw SQL, transaction effects, resource cost, failure behavior, and redacted terminal evidence together.

This prerelease remains experimental evaluation software. It is not production-ready and makes no backward-compatibility promise across prereleases.

## Included checked foundation

- Consumer Contract version 7 and Strict Profile version 2, including permanent diagnostics `PHT001` through `PHT006`.
- The installed `phpthis check` command, framework-owned maximum-level PHPStan configuration, repository guardrails, and mandatory application-owned automated behavior-test stage.
- Manual constructor composition, immutable request and response values, bounded PHP runtime request ingestion, exact-class error mapping, validated response cookies, optional lazy native-file sessions, and explicit response emission.
- Indexed literal and bounded typed routes with immutable typed parameter delivery.
- A thin PDO transport with finite compile-time-constant SQL, unique named bindings, explicit transactions, query budgets, and bounded redacted query traces.
- An independently packageable health-only application skeleton with project-owned AI context, explicit terminal summaries, and a complete profile-plus-behavior-test gate.

## Core additions

- Deterministic indexed routing for explicit paths containing at most two full-segment `positive-int` or `token` parameters, with immutable type-specific delivery, literal precedence, and construction-time rejection of overlapping declarations.
- At most one normalized flat upload entry at the typed bounded multipart request boundary and one concrete local-file response emitted in chunks of at most 8,192 bytes, with explicit framing and emission failures.
- An observable query-budget exceeded state for application-owned bounded terminal summaries.
- Stricter response-header validation and a fixed generic unknown-failure response using `Cache-Control: private, no-store`.

The file-transfer core owns only HTTP ingestion values and local-file emission. Applications continue to own upload fields and limits, provenance, storage, cleanup, authorization, retention, representation headers, filesystem topology, and deployment evidence. Byte ranges remain unsupported.

## Application-owned patterns and evidence

The source repository adds reviewed, executable evidence for the following patterns, while the framework package carries the corresponding contracts and AI guidance:

- explicit authentication, tenant resolution, and action authorization in that order;
- one-time parsing of external `mixed` input into operation-specific final readonly values;
- finite raw SQLite data paths with explicit bindings, bounded reads, query budgets, traces, and scale-sensitive tests;
- one correlated, redacted terminal request-summary attempt at the application front controller;
- transactionally published SQLite durable jobs with bounded envelopes, leases, retries, crash recovery, and an idempotent demonstrated effect;
- one explicit application console, finite command dispatch, and externally invoked scheduled passes;
- forward-only SQLite migrations with a finite unrolled manifest, checksummed ledger, per-migration transactions, separate authority, and a same-host nonblocking lock; and
- an application-owned Redis document cache and owner-token schedule lease with explicit topology, failure, stale-refill, renewal, and non-fencing limits.

These are application patterns, not framework services. ADR 029 records the ownership exit for every evaluated capability. Only bounded multiple typed routes and bounded HTTP file-transfer primitives enter core. The integrated `POST /accounts/{account_id:positive-int}/users` proof, its identity-separated `account_users` relation, welcome-job publication, SQLite schema, policies, and terminal summary remain replaceable application evidence.

## Consumer Contract upgrade

Alpha 2 advances Consumer Contract version 3 to version 7. Strict Profile version 2 and permanent diagnostics `PHT001` through `PHT006` are unchanged. Consumers upgrading directly from Alpha 1 apply the versioned changes in order.

### Contract version 4: bounded multiple typed routes

- Existing literal routes and one trailing `{name:positive-int}` declarations remain valid.
- Routes may now contain at most two full-segment parameters using `positive-int` or `token`.
- Replace direct calls to `Route::literalPrefix()` and `Route::parameterName()` with `Route::segments()` and, when needed, `Route::parameterCount()`.
- Read validated values through `PathParameters::positiveInteger()` or `PathParameters::token()` and immediately wrap them in route-specific identifiers.
- Resolve any declaration newly rejected by deterministic overlap checks; registration order and parameter-type preference do not resolve ambiguity.

### Contract version 5: application-owned terminal request summaries

- Replace `UnknownFailureBoundary::logAndRespond(Throwable)` and its global error-log path with `UnknownFailureBoundary::respond()` called without an argument inside one visible application-owned terminal coordinator. Retain the `Throwable` only for the coordinator's class-only unknown-failure summary.
- Generate and propagate one 128-bit lowercase-hex `X-Request-ID`.
- Register every request-scoped database connection that can execute anywhere in the coordinator path, even when the selected route does not use it, within a finite set of at most eight distinctly named sources. Every source owns its own `QueryBudget` and `QueryTrace`; do not share request-wide observation state.
- Attempt exactly one bounded redacted summary after response selection.
- Add the required success, mapped-failure, denial, unknown-failure, identifier, query-bound, duplicate-query, redaction, and throwing-sink evidence. A sink attempt does not establish durable delivery.

Version 5 adds no framework logger, middleware, event pipeline, sink, coordinator, service locator, discovery mechanism, or hidden database instrumentation.

### Contract version 6: bounded file transfers

- Update the front controller and terminal coordinator to pass parsed fields and files explicitly to `RequestBoundary`; configure a multipart cap only when adopting uploads.
- Preserve `Response::$fileBody` through every application response-copy path.
- Remove case-insensitively duplicated response headers and any header value containing an ASCII control byte or DEL.
- Keep all prior output outside `ResponseEmitter::emit()`: it now rejects already-sent headers for ordinary string responses as well as local-file responses with `ResponseEmissionFailed(true)`. Attempt one generic fallback only when `responseStarted` is `false`; never emit a replacement after it is `true`.
- Record and test the adopting application's exact upload, storage, cleanup, response, failure, deployment, and real-SAPI policies.

Existing non-multipart requests and ordinary string responses retain their behavior through optional arguments. Version 6 adds no storage facade, automatic persistence or cleanup, client-metadata trust, content processing, remote-object abstraction, or range implementation.

### Contract version 7: reviewed runtime and failure policy

- Framework and skeleton Composer requirements narrow from `^8.4` to `~8.4.0`, so the supported runtime is exactly PHP 8.4.x.
- PHP 8.4 applications require no source change for the runtime correction and must rerun their complete check.
- PHP 8.5 and later are unsupported until a separate compatibility decision and complete CI evidence exist. Those consumers cannot upgrade to Alpha 2. Alpha 1's broader constraint admitted PHP 8.5 without review and is not a support claim; remaining there requires independent consumer validation. Do not bypass Composer's platform check.
- Update exact generic unknown-failure response assertions from `Cache-Control: no-store` to `Cache-Control: private, no-store`.
- ADR 029's account-scoped Create path, `account_users` migration, and consumer-profile harness remain optional application evidence rather than required consumer schema or policy.

Applications must merge contract changes deliberately into their project-owned `AGENTS.md`, `.ai/` context, composition, and tests. Updating PHPThis does not authorize overwriting application-owned context.

## Runtime and portability boundary

The supported framework runtime is PHP 8.4.x. The base PDO transport continues to be certified separately against SQLite, MySQL 8.4, and PostgreSQL 17. That certification does not make application SQL, DDL, locking, returned values, errors, or transaction behavior portable between engines.

The accepted migration, durable-job, finite-data-path, and integrated consumer proofs are SQLite-specific. The Redis proof is application-owned and requires separately recorded client, server, topology, capacity, eviction, timeout, outage, and coordination decisions. PHPThis provides no core migration, queue, cache, lock, lease, or scheduler runtime.

## Permanent exclusions

Alpha 2 does not add or permit:

- an ORM, Active Record, model or repository layer, lazy loading, query builder, generated SQL, binding or placeholder helper, generic paginator, or dialect abstraction;
- implicit or global scopes, observer magic, automatic discovery, reflection-based wiring, a service container, generic facade, global helper, macro system, dynamic proxy, or autowiring mechanism;
- a generic authentication, identity, tenant, authorization, middleware, validation, session-security, logging, cache, queue, worker, scheduler, migration, storage, filesystem, or distributed-lock abstraction;
- hidden database calls, database work inside loops, hidden retries, automatic cache refill or invalidation, automatic job discovery, after-commit callbacks, inferred migration rollback, HTTP-startup migrations, or exactly-once external-effect claims; or
- production readiness, a support or security-response SLA, complete CRUD, Update or Delete evidence, portable application SQL or DDL, Redis failover guarantees, filesystem guarantees, deployment certification, or capacity certification.

Consumers remain responsible for their concrete identity, tenancy, authorization, schema, database grants and plans, concurrency, conflict behavior, migration recovery, external effects, caching consistency, coordination topology, file ownership, deployment, observability destination, and operational evidence.
